# Agent Deployment Guide

Complete guide for deploying an Atlax agent on any supported platform.

---

## Overview

The agent runs on a machine behind NAT/CGNAT and:
- Dials out to the relay over mTLS 1.3 (no inbound ports required)
- Receives stream requests through the multiplexed tunnel
- Forwards traffic to local services (Docker containers, VMs, bare-metal apps)
- Auto-reconnects with exponential backoff if the connection drops

```
┌──────────────────────────────────┐
│       AGENT (behind CGNAT)       │
│                                  │
│  atlax-agent ──► relay:8443      │  (outbound mTLS)
│       │                          │
│       ├── http    → 127.0.0.1:3000  (Next.js)
│       ├── api     → 127.0.0.1:7070  (Go API)
│       ├── smb     → 127.0.0.1:445   (Samba)
│       └── ssh     → 127.0.0.1:22    (SSH)
│                                  │
└──────────────────────────────────┘
```

## Network Requirements

The agent only makes **outbound** connections:

| Direction | Port | Protocol | Destination | Purpose |
|-----------|------|----------|-------------|---------|
| Outbound | 8443 | TCP/TLS | Relay IP | mTLS tunnel |
| Outbound | 443 | TCP | Internet | Cert renewal, updates |
| Outbound | 53 | UDP/TCP | DNS | Name resolution |
| Loopback | varies | TCP | 127.0.0.1 | Local service forwarding |

**No inbound ports are required.** This is why Atlax works behind CGNAT.

## Interactive Setup

```bash
ats setup agent
```

The wizard handles binary installation, config generation, cert copying, service registration, and verification.

## Manual Step-by-Step

### 1. System Preparation

```bash
# Create system group (controls file access)
sudo groupadd --system atlax

# Create service user (optional, runs the daemon)
sudo useradd --system --no-create-home --shell /usr/sbin/nologin -g atlax atlax

# Create directories
sudo mkdir -p /etc/atlax/certs
sudo chgrp -R atlax /etc/atlax
sudo chmod -R g+r /etc/atlax
sudo find /etc/atlax -name "*.key" -exec chmod 640 {} \;
```

### 2. Binary Installation

Same options as the relay — download, build, or copy:

```bash
VERSION="latest"
ARCH=$(uname -m | sed 's/x86_64/amd64/; s/aarch64/arm64/')
OS=$(uname -s | tr '[:upper:]' '[:lower:]')

curl -fsSL -o /tmp/atlax-agent \
  "https://github.com/atlasshare/atlax/releases/download/${VERSION}/atlax-agent-${OS}-${ARCH}"

sudo install -m 755 /tmp/atlax-agent /usr/local/bin/atlax-agent
```

### 3. Certificate Setup

Copy the agent certs (generated by `ats certs init` or `ats certs issue`):

```bash
sudo cp agent.crt agent.key relay-ca.crt /etc/atlax/certs/
sudo chmod 600 /etc/atlax/certs/agent.key
sudo chown -R atlax:atlax /etc/atlax/certs/
```

Required files on the agent:

| File | Purpose |
|------|---------|
| `agent.crt` | Agent client cert (CN must match customer ID in relay config) |
| `agent.key` | Agent private key |
| `relay-ca.crt` | Relay CA cert (verifies the relay's TLS cert) |

**Critical:** The `CN` in `agent.crt` (e.g., `customer-dev-001`) must exactly match a customer `id` in the relay's `relay.yaml`. The relay extracts the customer identity from the mTLS peer certificate during the handshake.

### 4. Configuration

Create `/etc/atlax/agent.yaml`:

```yaml
relay:
  addr: "relay.example.com:8443"
  server_name: "relay.atlax.local"
  reconnect_interval: 5s
  reconnect_max_backoff: 300s
  reconnect_jitter: true
  keepalive_interval: 30s
  keepalive_timeout: 10s

tls:
  cert_file: /etc/atlax/certs/agent.crt
  key_file: /etc/atlax/certs/agent.key
  ca_file: /etc/atlax/certs/relay-ca.crt

services:
  - name: http
    local_addr: "127.0.0.1:3000"
    protocol: tcp
    description: "Web application"

  - name: api
    local_addr: "127.0.0.1:7070"
    protocol: tcp
    description: "API server"

logging:
  level: info
  format: json
```

**Configuration reference:**

| Field | Default | Description |
|-------|---------|-------------|
| `relay.addr` | — | Relay host:port (required) |
| `relay.server_name` | — | TLS server name for cert verification |
| `relay.reconnect_interval` | `5s` | Initial reconnect backoff |
| `relay.reconnect_max_backoff` | `300s` | Maximum backoff duration |
| `relay.reconnect_jitter` | `true` | Add jitter to prevent thundering herd |
| `relay.keepalive_interval` | `30s` | PING interval |
| `relay.keepalive_timeout` | `10s` | PONG timeout |
| `services[].name` | — | Must match `ports[].service` in relay config |
| `services[].local_addr` | — | Where to forward traffic locally |
| `services[].protocol` | `tcp` | Transport protocol |

### 5. Service Name Matching

**This is the most common source of errors.** The service name on the agent must exactly match the service name on the relay for the same customer.

```
Relay config (relay.yaml)               Agent config (agent.yaml)
─────────────────────────               ────────────────────────
customers:                              services:
  - id: customer-dev-001                  - name: http          ◄── MUST MATCH
    ports:                                  local_addr: 127.0.0.1:3000
      - port: 18080
        service: http         ◄── MUST MATCH
```

When a client connects to relay port 18080:
1. Relay looks up port 18080 → customer `customer-dev-001`, service `http`
2. Relay sends `STREAM_OPEN` with payload `http` to the agent
3. Agent receives `http`, looks it up in `services[]`, finds `127.0.0.1:3000`
4. Agent opens a TCP connection to `127.0.0.1:3000` and bridges the stream

### 6. System Service

**systemd (Linux):**

```ini
# /etc/systemd/system/atlax-agent.service
[Unit]
Description=Atlax Agent
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=atlax
Group=atlax
ExecStart=/usr/local/bin/atlax-agent -config /etc/atlax/agent.yaml
Restart=on-failure
RestartSec=5s
WatchdogSec=30s
LimitNOFILE=65536

NoNewPrivileges=yes
ProtectSystem=strict
ProtectHome=yes
PrivateTmp=yes
MemoryDenyWriteExecute=yes
LockPersonality=yes
ReadOnlyPaths=/etc/atlax

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now atlax-agent
```

**launchd (macOS):**

```xml
<!-- /Library/LaunchDaemons/io.atlax.agent.plist -->
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>io.atlax.agent</string>
    <key>ProgramArguments</key>
    <array>
        <string>/usr/local/bin/atlax-agent</string>
        <string>-config</string>
        <string>/usr/local/etc/atlax/agent.yaml</string>
    </array>
    <key>RunAtLoad</key>
    <true/>
    <key>KeepAlive</key>
    <true/>
</dict>
</plist>
```

**Windows Service:**

```powershell
# Install with NSSM (Non-Sucking Service Manager)
nssm install atlax-agent "C:\Program Files\atlax\atlax-agent.exe" -config "C:\ProgramData\atlax\agent.yaml"
nssm set atlax-agent Start SERVICE_AUTO_START
nssm start atlax-agent
```

### 7. Verification

```bash
# Service running
systemctl status atlax-agent

# Check agent logs for connection
journalctl -u atlax-agent --since "5 minutes ago"
# Look for: "connected to relay"

# Verify from the relay side
ssh relay-host "curl -s http://localhost:9090/metrics | grep atlax_relay_agents_connected"
# → atlax_relay_agents_connected 1

# Full diagnostics
ats health
# Select: "Agent"
```

## Docker Services

A common pattern is running local services in Docker and having the agent forward to them:

```yaml
# docker-compose.yml
services:
  web:
    build: .
    ports:
      - "127.0.0.1:3000:3000"    # Bind to localhost only
    restart: unless-stopped

  api:
    build: ./api
    ports:
      - "127.0.0.1:7070:7070"
    restart: unless-stopped
```

```yaml
# agent.yaml (services section)
services:
  - name: http
    local_addr: "127.0.0.1:3000"
    protocol: tcp
  - name: api
    local_addr: "127.0.0.1:7070"
    protocol: tcp
```

**Always bind Docker ports to `127.0.0.1`** so only the agent can reach them — not the public internet.

## Reconnection Behavior

The agent auto-reconnects when the tunnel drops:

1. **Initial backoff:** 5 seconds
2. **Exponential increase:** 5s → 10s → 20s → 40s → ...
3. **Max backoff:** 300 seconds (5 minutes)
4. **Jitter:** 0-50% random delay added to prevent thundering herd
5. **On success:** backoff resets to initial value

No operator intervention is needed for temporary network issues, relay restarts, or VPS reboots.

## Resource Usage

The agent is lightweight:

| Metric | Value |
|--------|-------|
| Memory | ~30-50 MB |
| CPU | Negligible (idle unless streaming) |
| Network | mTLS keepalive PINGs every 30s (~200 bytes each) |
| Disk | Binary (~7 MB) + config + certs |

## Troubleshooting

| Symptom | Cause | Fix |
|---------|-------|-----|
| "cannot connect to relay" | Outbound 8443 blocked | Check egress firewall/NAT rules |
| "TLS handshake failed" | Cert chain broken | Verify agent cert signed by customer-ca, relay cert signed by relay-ca |
| "unknown service" in relay logs | Name mismatch | Ensure `services[].name` in agent matches `ports[].service` in relay |
| Agent connects but traffic doesn't flow | Local service down | Check `curl http://127.0.0.1:<port>` on the agent host |
| Frequent reconnections | Network instability | Check ISP, increase `keepalive_timeout` |
| "customer not found" | Cert CN doesn't match relay config | Verify cert CN matches a customer `id` |
